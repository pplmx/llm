# OpenAI-compat `logit_bias` (Tier 3 #38)

## Source

`ChatCompletionRequest` in `serving/schemas.py` does not declare
`logit_bias`. OpenAI's chat completion spec exposes it as:

> "Modify the likelihood of specified tokens appearing in the
> completion. Accepts a JSON object that maps tokens (specified by
> their token ID in the GPT tokenizer) to an associated bias value
> from -100 to 100. Mathematically, the bias is added to the logits
> generated by the model prior to sampling."

The parameter is widely used by OpenAI clients (the official Python
SDK sets it on every request that wants deterministic token bans /
exclusives, e.g. force a specific format prefix). Silently dropping
it breaks those clients.

## Description

Plumb `logit_bias` end-to-end so an OpenAI client can suppress
specific tokens (negative bias) or force a near-exclusive selection
of specific tokens (large positive bias, e.g. `100`). The
implementation **adds** the bias to the affected logits before
sampling — same mathematical placement as HuggingFace's
`LogitsProcessor` chain (`forced_bos_token_id`, `forced_eos_token_id`,
`NoBadWordsLogitsProcessor`).

The bias is applied **after** the penalty helpers (repetition,
frequency, presence). Rationale: a penalty subtracts to discourage
repetition, and the bias is a user-intent override — applying it
last lets the bias override any natural penalty the model would
otherwise impose. (OpenAI's reference implementation applies the
bias at the same final-logit stage; ordering with their internal
penalties is equivalent.)

## Design

- `src/llm/generation/sampling.py`:
    - New `apply_logit_bias(logits, logit_bias) -> logits`. Adds
      `logit_bias[tid]` to each affected token's logit via
      `index_add_` (or `scatter_add_` with positive bias + clone).
      Returns a new 1D tensor — input is never mutated.
    - No-op when `logit_bias` is `None` or empty.
    - Token ids outside `[0, vocab_size)` are silently ignored
      (matches `apply_presence_penalty` / `apply_frequency_penalty`).

- `src/llm/generation/backends.py`:
    - `GenerationConfig` gains `logit_bias: dict[int, float] | None = None`.
      Uses `field(default_factory=dict)` so the frozen dataclass
      stays immutable across instances.
    - All three backends forward the field (default `None` is a
      no-op so existing tests pass unchanged).

- `src/llm/generation/eager.py`:
    - `stream_generate` / `generate` / `batch_generate` accept
      `logit_bias`. `stream_generate` applies it per-token after
      `presence_penalty`. `batch_generate` applies it per-row.

- `src/llm/generation/speculative.py`:
    - `speculative_generate` accepts and applies `logit_bias` to
      the draft logits.

- `src/llm/serving/generation_service.py`:
    - `generate` / `stream` / `batch_generate` /
      `_generation_config` all forward `logit_bias`.

- `src/llm/serving/routers/chat.py`:
    - Pass `logit_bias=request.logit_bias` to `_sync_generate` and
      the streaming generator.

- `src/llm/serving/routers/generate.py`:
    - `GenerationRequest` gains `logit_bias: dict[str, float] | None = None`.
      The router forwards it.

- `src/llm/serving/schemas.py`:
    - `ChatCompletionRequest.logit_bias: dict[str, float] | None = None`.
      JSON keys are strings per the spec; the helper coerces them
      to `int` at the boundary.
    - `BatchGenerationRequest.logit_bias: dict[str, float] | None = None`
      (matches `GenerationRequest`).

## Acceptance criteria

- [x] `apply_logit_bias` is exposed and tested: empty/`None` is a
      no-op; non-empty adds the bias to each affected logit;
      ignores ids outside vocab; does not mutate input.
- [x] `GenerationConfig` includes `logit_bias`. Round-trip via
      the dataclass works.
- [x] All three backends forward `logit_bias`. Existing tests
      still pass with the new parameter at its default of `None`.
- [x] `ChatCompletionRequest.logit_bias` is accepted and forwarded
      to the service as a `dict[str, float]` (JSON keys are
      strings; the helper coerces to int at the logit-stage).
- [x] End-to-end: sending `logit_bias={"42": 100.0}` to
      `/v1/chat/completions` reaches the service's
      `generate(prompt=..., logit_bias={"42": 100.0})`.

## Out of scope (intentional)

- **`logit_bias` value range validation** — OpenAI's docs say
  `[-100, 100]` but the spec doesn't reject out-of-range values
  with a 422; the bias is just clamped by the model in practice.
  We pass through whatever the client sends. Schema-side clamping
  is a follow-up if we ever want a strict-mode contract.
- **`logprobs` / `top_logprobs`** — a separate OpenAI-compat
  feature (returning the top-K log probabilities of the sampled
  tokens in the response). Bigger slice; involves response-shape
  changes.
- **`n` (multiple completions)** — OpenAI-compat but requires
  running the generation loop N times in parallel or in sequence,
  touching every backend. Separate slice.
- **`tools` / function calling** — large feature; not a
  per-parameter gap.
